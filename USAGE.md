## Швидкий старт

Потрібен Python 3.10+.

```bash
# 1. Встановити залежність (одна, тільки cryptography)
pip install -r requirements.txt

# 2. Запустити
python main.py
```

При **першому** запуску програма попросить **створити master password**.
При наступних запусках — **ввести** його для розблокування.

> 💡 Master password — це єдиний пароль, який тобі треба запам'ятати. Він
> не зберігається у БД; якщо забудеш — всі дані буде втрачено назавжди.

## Master password

### Перший запуск

```
Master password ще не задано. Створіть його зараз.
Новий master password: ●●●●●●●●
Підтвердіть master password: ●●●●●●●●
Master password встановлено.
```

Після цього програма одразу відкриє меню.

### Подальші запуски

```
Master password: ●●●●●●●●
```

- Вірно → доступ до меню.
- Невірно → `Невірний пароль. Залишилось спроб: 4.` і так до 5 спроб.
- Після 5 невдалих спроб програма закриється з кодом `1` і повідомленням
  `Перевищено кількість спроб.`.

> Поле введення майстер-пароля **не показує символи** (через `getpass`).
> На деяких терміналах курсор стоїть на місці — це нормально.

### Чому це безпечно

Master password ніколи не зберігається. У БД лежать тільки:
- `salt` (16 випадкових байт)
- `verifier` — короткий зашифрований токен
- `kdf_version` — ідентифікатор KDF (`argon2id-v1`)

При вході програма виводить ключ із master password + salt через
**Argon2id** (memory_cost=19 MiB, time_cost=2, parallelism=1 — OWASP 2024),
а потім намагається розшифрувати verifier. Якщо вдалося — пароль вірний;
якщо ні — не пускає до меню.

#### Чому Argon2id а не PBKDF2

Argon2id — переможець Password Hashing Competition 2015. На відміну від
PBKDF2 він *memory-hard*: кожна спроба вимагає повних 19 MiB RAM,
що різко знижує ефективність GPU/ASIC brute-force. PBKDF2 хороше роз-
паралелюється на GPU з тисячами ядер — Argon2id ні.

#### Міграція зі старої БД

Якщо твоя `users.db` була створена попередніми версіями програми
(до цього релізу) — вона використовує PBKDF2-HMAC-SHA256 (480 000
ітерацій). Такі БД **відкриваються без жодних ручних дій** —
програма автоматично детектує старий формат і після входу покаже:

```
Увага: ця БД використовує старий KDF (PBKDF2). Змініть master password
(пункт 10) щоб мігрувати на Argon2id — нічого не доведеться вводити повторно.
```

Щоб мігрувати: меню → `10) Змінити master password`. Можна ввести
той же самий master password як старий і новий — KDF все одно
перемкнеться на Argon2id, бо функція робить повне перешифрування записів
свіжим salt + KDF в одній SQLite-транзакції.

## Меню та всі команди

Після входу побачиш:

```
=== Password Manager ===
1)  Додати акаунт
2)  Знайти акаунт
3)  Показати всі акаунти
4)  Видалити акаунт
5)  Оновити пароль
6)  Експорт у JSON
7)  Імпорт з JSON
8)  Пошук по login/email
9)  Вихід
10) Змінити master password
11) Згенерувати пароль

Виберіть пункт: _
```

Введи номер і Enter.

### 1) Додати акаунт

Послідовно запитає три поля:

```
Login: alice
Email: alice@example.com
Password (або 'g' щоб згенерувати): ●●●●●●●●
Створено:
  id=1 login='alice' email='alice@example.com' password='p@ssw0rd!' created_at=2026-04-26 18:30:00
```

Правила:
- `login` — обов'язковий, **унікальний**. Спроба додати дублікат → 
  `Помилка: login 'alice' already exists`.
- `email` — необов'язковий (можна лишити пустим Enter).
- `password` — будь-який непорожній рядок, шифрується одразу. Або введи `g` (без лапок) — відкриється інтерактивний генератор (див. пункт 10), і отриманий пароль відразу піде в поле password.
- Пустий password відхилється з проханням повторити ввід.

### 2) Знайти акаунт

Точне співпадіння за `login`:

```
Login для пошуку: alice
  id=1 login='alice' email='alice@example.com' password='p@ssw0rd!' created_at=...
```

Якщо немає — `Не знайдено.`

> Для часткового пошуку використовуй пункт **8** (підрядок по login + email).

### 3) Показати всі акаунти

Виводить **усі** записи з **розшифрованими** паролями:

```
  [1] id=1 login='alice' email='alice@example.com' password='p1' created_at=...
  [2] id=2 login='bob'   email='bob@corp.io'      password='p2' created_at=...
```

**Формат рядка**:
- `[N]` — порядковий номер у списку (завжди починається з `1`, без пропусків).
- `id=N` — стабільний DB id запису. Ніколи не перевикористовується після видалення (це SQLite `AUTOINCREMENT`-семантика). Тому якщо ти створив запис, видалив, створив знов — `[1] id=2` (рядок перший, але DB id вже 2).

Якщо БД порожня — `(порожньо)`.

### 4) Видалити акаунт

```
Login для видалення: alice
Видалено.
```

Видаляє назавжди — підтвердження не запитує. Якщо не знайдено — 
`Не знайдено.`

### 5) Оновити пароль

```
Login: alice
Новий password (або 'g' щоб згенерувати): ●●●●●●●●
Оновлено.
```

Якщо логіну немає — `Не знайдено.` (без змін). Так само як в пункті 1, можна ввести `g` щоб згенерувати новий.

### 6) Експорт у JSON

```
Шлях до файлу експорту [export.json]: ~/Desktop/passwords.json
Експортовано 42 акаунтів у ~/Desktop/passwords.json.
```

- Натисни Enter без тексту → буде створено `export.json` у поточній теці.
- `~` розгортається в твою домашню директорію.
- `$VAR` розгортається у значення змінної середовища (наприклад, `$HOME/dump.json`).
- **Якщо батьківські теки не існують — вони будуть створені автоматично.**
- Якщо немає прав на запис — побачиш `Помилка експорту: [Errno 13] Permission denied: …`.

#### Формат JSON

```json
[
  {
    "id": 1,
    "login": "alice",
    "email": "alice@example.com",
    "password": "p@ssw0rd!",
    "created_at": "2026-04-26 18:30:00"
  },
  {
    "id": 2,
    "login": "bob",
    "email": "bob@corp.io",
    "password": "hunter2",
    "created_at": "2026-04-26 18:31:00"
  }
]
```

> ⚠️ **Експорт містить паролі у відкритому вигляді.** Це задумано (так
> можна відновити дані навіть без програми). Зберігай файл у безпечному
> місці й видали після використання: `shred -u export.json` (Linux) або
> `rm -P export.json` (macOS).

### 7) Імпорт з JSON

```
Шлях до файлу імпорту: ~/Downloads/old_passwords.json
Імпортовано 15 нових акаунтів (дублікати пропущено).
```

- Файл має бути JSON-масивом об'єктів з полями `login`, `email`, `password` (як вище).
- Записи з `login`, який уже є в БД, **пропускаються** (не перезаписуються).
- Паролі з файлу **перешифровуються** твоїм поточним master password,
  тож можна імпортувати дамп зі старою БД у нову з іншим master.
- Якщо файл не існує → `Помилка імпорту: [Errno 2] No such file or directory: …`.
- Якщо JSON битий → `Помилка імпорту: Expecting value: line 1 column 1 (char 0)` тощо.
- Якщо вміст не масив → `Помилка імпорту: import file must contain a JSON array of objects`.

#### Як міграція з іншої БД

```
# у старій БД
6) → export.json
9) Вихід

# в новому середовищі (інший master password — це ок)
$ python main.py
master password ще не задано. Створіть.
7) → export.json
3) → побачиш всі імпортовані акаунти з оригінальними паролями
```

### 8) Пошук по login/email

Підрядковий пошук (case-insensitive):

```
Пошуковий запит: corp
  [1] id=3 login='bob' email='bob@corp.io' password='pb' created_at=...
```

Знаходить будь-який запис, де `corp` зустрічається у `login` **або** `email`.

### 9) Вихід

```
До побачення.
```

Завершує програму з кодом `0`.

> Натиснення `Ctrl+C` у будь-який момент — теж акуратно вийде.

### 10) Змінити master password

Перешифровує **всі** збережені паролі під новим ключем в одній SQLite-транзакції.
Одночасно оновлює `salt` і verifier-токен. Або все оновиться, або нічого —
без напів-застосованого стану.

```
Поточний master password: ●●●●●●●●
Новий master password: ●●●●●●●●●●●
Підтвердіть новий master password: ●●●●●●●●●●●
Master password змінено. Перешифровано 42 акаунтів під новим ключем.
```

Типові помилки:

| Повідомлення | Причина |
|---|---|
| `Помилка: old master password is incorrect` | Невірно введено поточний master |
| `Новий master password не може бути порожнім.` | Пустий новий пароль |
| `Не співпадає, спробуйте ще раз.` | Підтвердження не збігається з новим паролем |

> Після успіху операції програма залишається офвлоченою під новим master — не треба вводити його ще раз.

### 11) Згенерувати пароль

Створює криптографічно випадковий пароль через `secrets.SystemRandom`
(базується на OS CSPRNG, а не на стандартному `random`).

```
Довжина [20]: 24
Нижній регістр (a-z)? [Y/n]: 
Верхній регістр (A-Z)? [Y/n]: 
Цифри (0-9)? [Y/n]: 
Символи (!@#…)? [Y/n]: 
Згенеровано: 2Oasx,FUc(RP6aV{)Qs>/+v2
```

- Порожній ввід на будь-яке питання — включення замовчуванням (`Y`).
- Довжина за замовчуванням — **20** (приблизно 130 біт ентропії на повному алфавіті з 70+ символів).
- Гарантовано містить **хоча б один символ з кожного включеного класу**.
- Допустима довжина: 4 … 4096. Символи за замовчуванням: `!@#$%^&*()-_=+[]{};:,.<>/?` (без лапок, бекслешів і пробілів, щоб пароль безпечно копіювався через shell і веб-форми).

#### Виклик з всередини других пунктів

Під час `1) Додати акаунт` або `5) Оновити пароль` можна ввести просто `g` в полі password
— відкриється той самий генератор, і результат відразу піде в поле password — не треба копіювати вручну.

```
Login: github
Email: me@example.com
Password (або 'g' щоб згенерувати): g
Довжина [20]: 
Нижній регістр (a-z)? [Y/n]: 
Верхній регістр (A-Z)? [Y/n]: 
Цифри (0-9)? [Y/n]: 
Символи (!@#…)? [Y/n]: 
Згенеровано: K[]b9.r…
Створено:
  id=1 login='github' email='me@example.com' password='K[]b9.r…' created_at=…
```

Типові помилки:

| Повідомлення | Причина |
|---|---|
| `Невірна довжина: 'foo'.` | Ти ввів не число |
| `Помилка генератора: length must be between 4 and 4096` | Довжина поза діапазоном |
| `Помилка генератора: at least one character class must be enabled` | Ти відповів `n` на всі 4 питання про класи |

## Файли програми

У теці звідки ти запустив `python main.py` з'являться:

| Файл | Призначення | Можна видаляти? |
|---|---|---|
| `users.db` | SQLite з усіма (зашифрованими) паролями + master salt | **НІ** — втратиш все |
| `export.json` | Дамп з пункту 6 | Так, після використання |
| `__pycache__/` | Кеш Python | Так, в будь-який момент |

> `users.db` і `*.json` уже у `.gitignore` — випадково не закомітиш.

### Перенесення на інший комп'ютер

Просто скопіюй `users.db` — він самодостатній. Master password ти вже
знаєш, ключ деривується з нього + salt, що лежить у файлі.

## Що робити, якщо…

### …забув master password
Відновити неможливо — це навмисно (вся безпека на цьому й тримається).
Видали `users.db` і починай з нуля. Якщо є `export.json` зі старими
даними — їх можна імпортувати у новій БД з новим master.

### …потрібно змінити master password
Використовуй пункт меню **10) Змінити master password** — він перешифровує
всі записи під новим ключем атомарно в одній SQLite-транзакції.

### …`Експортовано 0 акаунтів`
Просто немає жодного запису. Додай через пункт 1.

### …`Помилка експорту: [Errno 13] Permission denied`
Немає прав на запис у вибрану теку. Спробуй `~/Desktop/export.json` або
поточну теку (просто `Enter`).

### …`Помилка імпорту: Expecting value`
JSON-файл битий. Перевір, що вміст починається з `[` і закінчується `]`,
без BOM і без зайвих ком.

### …хочу видалити всі дані
```
rm users.db
```
Готово. Наступний запуск попросить новий master password і почне з нуля.

### …хочу безпечно видалити експорт
- Linux: `shred -u export.json`
- macOS: `rm -P export.json`
- Windows: `cipher /w:.` (для всього диска це повільно — простіше зашифрувати або не експортувати взагалі)

### …програма раптом просить master password посеред сесії
Це авто-блокування за бездіяльністю — якщо нічого не натискати кілька
хвилин, ключ скидається з пам'яті, і перед наступною операцією треба
ввести master заново. Деталі (тайм-аут, env-змінна, як вимкнути) — у
розділі [Авто-блокування за бездіяльністю](#авто-блокування-за-бездіяльністю).

## Програмний API (для скриптів)

Якщо хочеш викликати функції з власного коду:

```python
from password_manager import PasswordManager

mgr = PasswordManager("users.db")

if not mgr.has_master_password():
    mgr.set_master_password("super-secret")
else:
    if not mgr.verify_master_password("super-secret"):
        raise SystemExit("wrong master password")

# CRUD
mgr.create_user("alice", "alice@example.com", "p@ssw0rd!")
record = mgr.get_user("alice")
print(record.password)         # 'p@ssw0rd!'
print(record.email)            # 'alice@example.com'

for r in mgr.list_users():
    print(r.login, r.password)

mgr.update_password("alice", "new-secret")
mgr.delete_user("alice")

# Search (substring, case-insensitive, login OR email)
hits = mgr.search("corp")

# Import / Export
mgr.export_to_json("~/Desktop/dump.json")        # ~ розгортається
mgr.import_from_json("~/Desktop/dump.json")      # дублікати пропускаються

# Змінити master password — перешифровує всі записи атомарно
rotated = mgr.change_master_password("super-secret", "new-master-2025")
print(f"Перешифровано {rotated} акаунтів")

# Lock (забути ключ; наступна операція кине RuntimeError, поки не verify)
mgr.lock()
```

### Генератор паролів

```python
from password_manager import PasswordPolicy, generate_password

# Дефолт: 20 символів, всі 4 класи (a-z, A-Z, 0-9, !@#$%^&*()-_=+[]{};:,.<>/?)
pw = generate_password()

# Кастомна політика
policy = PasswordPolicy(
    length=32,
    use_lower=True,
    use_upper=True,
    use_digits=True,
    use_symbols=False,        # тільки літери + цифри
)
pw = generate_password(policy)

# Кастомний набір символів
policy = PasswordPolicy(length=16, symbols="!@#")
pw = generate_password(policy)
```

Гарантії:
- Джерело випадковості — `secrets.SystemRandom` (OS CSPRNG, не `random`).
- Якщо ввімкнено N класів, результат МІСТИТЬ хоча б один символ з кожного.
- Довжина від 4 до 4096.
- `ValueError` — якщо довжина поза діапазоном або всі класи вимкнено.


### Винятки

| Виняток | Коли |
|---|---|
| `RuntimeError("manager is locked …")` | Виклик мутатора без `verify_master_password` |
| `RuntimeError("master password already set")` | `set_master_password` на вже ініціалізованій БД |
| `ValueError("login 'x' already exists")` | `create_user` з існуючим логіном |
| `ValueError("master password must not be empty")` | Пустий master |
| `ValueError("new master password must not be empty")` | `change_master_password` з пустим новим |
| `ValueError("old master password is incorrect")` | `change_master_password` з неправильним старим |
| `ValueError("login must not be empty")` | Пустий login |
| `OSError` | Проблема з файлом експорту/імпорту |
| `ValueError("import file must contain a JSON array of objects")` | JSON не є масивом |

## Безпека: що захищене, а що ні

### ✅ Захищене
- **Паролі на диску** — Fernet (AES-128-CBC + HMAC-SHA256). Перевірено: 
  `strings users.db | grep <твій-пароль>` не знаходить нічого.
- **Master password не зберігається** — у БД тільки salt, verifier і KDF-ідентифікатор.
- **Argon2id (m=19 MiB, t=2, p=1)** — memory-hard KDF, стійкий до GPU/ASIC brute-force.
  Старі БД з PBKDF2 (480k ітерацій) мігруються автоматично при зміні master.
- **Унікальний salt** на кожну БД — rainbow-table атаки не працюють.
- **Дублікати login** на рівні БД (UNIQUE constraint).

### ⚠️ Не захищене (за дизайном)
- **`export.json` — це плейнтекст**. Це фіча для відновлення/міграції,
  але файл потрібно стерти після використання.
- **Email і login зберігаються відкрито** — зашифровано тільки `password`.
- **RAM не очищається** — якщо хтось зніме дамп пам'яті процесу, master
  password можна знайти.
- **Локальний доступ до файлу** = повний компроміс після злому master.
  Програма не захищає від root на твоєму комп'ютері.
- **Авто-блокування за бездіяльністю є** (за замовч. 5 хв) — деталі нижче.
  Це зменшує ризик "запустив і пішов на каву", але не замінює блокування
  екрана ОС.

### Рекомендації
- Master password — щонайменше 16 символів, унікальний, не використовуй
  його ніде більше.
- `users.db` — на зашифрованому диску (LUKS / FileVault / BitLocker).
- `export.json` — створюй тільки коли треба, видаляй одразу.
- Регулярні бекапи `users.db` (він самодостатній).

## Авто-блокування за бездіяльністю

Програма автоматично "забуває" розшифрований ключ після `N` секунд без
активності. Перед наступною дією у меню вона знов запитає master password.

### Як це працює

- Таймер скидається тільки після **успішного завершення дії, що
  звертається до зашифрованої БД** (пункти 1–8, 10). Дії з винятку
  (див. нижче) **не скидають** таймер — тобто час відлічується від
  останньої БД-дії.
- Якщо між поверненням у меню та вибором наступного БД-пункту минуло
  більше за `N` секунд — менеджер скидає ключ (`PasswordManager.lock()`)
  і перед виконанням дії запитує master password.
- 5 невірних спроб поспіль → програма виходить з кодом `1`.
- Виняток: пункт **9) Вихід** і пункт **11) Згенерувати пароль**
  (генератор не звертається до зашифрованої БД) — auto-lock їх не
  активує і вони не продовжують сесію. Це навмисно: інакше idle-користувач
  міг би натиснути «11», щоб обійти повторну автентифікацію перед
  наступною БД-дією.

### Налаштування

Тайм-аут читається з env-змінної `PM_AUTO_LOCK_SECONDS` при старті:

| Значення | Поведінка |
|----------|-----------|
| (не задано) | **300 секунд (5 хвилин)** — за замовчуванням |
| `60` | блокування через 60 секунд бездіяльності |
| `0` | **вимкнено** — менеджер не блокується автоматично |
| `abc`, `-1`, тощо | помилкове → fallback до 300 (щоб typo не вимикав security) |

Приклади:

```bash
# 1 хвилина — для коротких сесій
PM_AUTO_LOCK_SECONDS=60 python main.py

# вимкнути auto-lock (не рекомендовано)
PM_AUTO_LOCK_SECONDS=0 python main.py

# 30 хвилин — для довгої роботи
PM_AUTO_LOCK_SECONDS=1800 python main.py
```

Можна додати в `~/.bashrc` / `~/.zshrc`:

```bash
export PM_AUTO_LOCK_SECONDS=120
```

### Обмеження

- Між моментом lock та переходом до прохання master, RAM-сторінки з
  попереднім ключем потенційно ще можуть лежати у вільних блоках. Python
  не дає очистити пам'ять детерміновано.
- Якщо процес заморожено (наприклад, `Ctrl+Z`), таймер відновлює відлік з
  останнього `time.monotonic()` тільки після `fg` — це коректна
  поведінка, бо `monotonic()` рахує реальний час.
